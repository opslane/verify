# WS7 Summary: Learner Race Spec Fixes + Follow-on Improvements

## What shipped

25 commits across 3 implementation plans. 203 tests (up from 181). E2E validated on Formbricks: **4 pass, 0 fail, 2 spec_unclear**.

### Plan 1: Learner Redesign + Race Condition + Spec Escalation
> `docs/plans/2026-03-19-learner-race-spec-fixes.md` — 9 tasks

**Problem:** Three issues found during e2e eval: (1) learner wrote behavioral conclusions like "ac2 is UNTESTABLE" that poisoned future runs, (2) parallel setup groups mutated the same DB rows causing race conditions, (3) no way to escalate spec inaccuracies — everything was pass or fail.

**What changed:**
- `spec_unclear` verdict type + judge prompt guidance + report "NEEDS HUMAN REVIEW" section + exit code 2
- Learner prompt rewritten to structured-only (SQL Corrections, Column Mappings, Required Fields, Timing)
- Learnings post-validator: deterministic defense-in-depth that strips unauthorized sections and banned patterns (MUST/NEVER/ALWAYS directives)
- Setup groups with DB conditions run sequentially; pure-UI groups run in parallel
- Setup writer prompt genericized (removed Formbricks-specific SQL examples)

### Plan 2: Browse Agent Execute-Only
> `docs/plans/2026-03-19-browse-agent-execute-only.md` — 4 tasks

**Problem:** ac6 timed out at 300s because the browse agent spent all its time searching the codebase for a feature list that doesn't exist on the billing page.

**What changed:**
- Browse agent prompt rewritten as pure executor: "navigate, screenshot, report reality"
- Tool restriction: `dangerouslySkipPermissions` → `allowedTools: ["Bash", "Read"]`
- Result: ac6 completes in 28-57s instead of timing out at 300s

### Plan 3: Setup Writer ORM Dispatch + Schema Capture
> `docs/plans/2026-03-19-setup-writer-orm-dispatch.md` — 8 tasks

**Problem:** Setup writer timed out at 240s exploring codebase to understand JSONB column shapes. Root cause: it had `dangerouslySkipPermissions` and an instruction to "read source code."

**What changed:**
- `pg_dump --schema-only` runs during index-app, stores `.verify/schema.sql` (generic, any Postgres project)
- Prisma JSONB type annotations extracted from `/// [TypeName]` comments, stored in `app.json` as `json_type_annotations`
- Shared `extractModelBody()` helper in prisma-parser (DRY refactor)
- Setup writer tool restriction: `dangerouslySkipPermissions` → `allowedTools: ["Bash", "Read"]`
- ORM-dispatched prompt selection via existing `detectORM()`: Prisma projects get `setup-writer-prisma.txt`
- `spec_unclear` added to `VALID_VERDICTS` in judge parser (was silently rejecting valid judge output)
- Judge prompt updated: "element completely absent from expected page" = spec_unclear signal
- Result: setup groups complete in 21-28s (down from 240s+)

## Areas of improvement still remaining

### P1 — Self-healing gap for JSONB field discovery

**Problem:** When the setup writer uses wrong JSONB field names (e.g., `status` instead of `subscriptionStatus`), the SQL succeeds but the app renders an error state. There's no SQL error for the learner to capture, so the self-healing loop doesn't fire. We had to manually seed learnings.md with `Required Fields` for the first run to work.

**Options:**
- A) Teach the learner to detect "app rendered error state" from judge verdicts + screenshots (if every AC in a group fails after setup, the setup was probably wrong)
- B) Pre-populate Required Fields from Prisma type annotations during index-app (we already extract `/// [OrganizationStripeBilling]` — we could also grep for the Zod schema fields and store them)
- C) Both — B for known types, A as fallback

**Impact:** Without this, first run on a new app with JSONB columns always fails. Second run works if someone manually adds the Required Fields entry.

### P1 — pg_dump version mismatch

**Problem:** `pg_dump` failed on Formbricks because local pg_dump is v16 but Formbricks runs Postgres 18 in Docker. `schema.sql` was not generated.

**Options:**
- Run `pg_dump` inside the Docker container if the DB is containerized
- Use `psql -c '\d+ tablename'` as a fallback (works regardless of version)
- Accept the gap — setup writer works fine with just app.json + learnings.md

### P2 — Planner retry adds 141s

**Problem:** The planner's first attempt often has validation errors (template variables in URLs), triggering a retry that adds ~141s. This is the biggest remaining contributor to pipeline time.

**Options:**
- Improve planner prompt to avoid template variables (add examples of correct vs wrong URLs)
- Give the planner access to app.json routes so it uses real URLs
- Accept — the retry mechanism works, it's just slow

### P2 — ac1 browse agent takes 55-87s for a simple sidebar check

**Problem:** ac1 just needs to check a sidebar trial alert. It should take 20s but sometimes takes 87s. The browse agent is likely taking multiple snapshots or clicking around even though the element is visible immediately.

**Options:**
- Add "if the element is visible in your first snapshot, report immediately" to the browse agent prompt
- This is similar to the execute-only fix but more aggressive — "one snapshot, one screenshot, report"

### P2 — TODOS.md Multi-ORM (Drizzle/TypeORM)

Already tracked in TODOS.md. Infrastructure is in place (ORM dispatch, detectORM). Adding a new ORM = new prompt file + optional schema parser.

### P3 — Eval suite for v2 stages

Already tracked in TODOS.md. No automated evals exist for the v2 pipeline stages. E2E on Formbricks is the only validation path. Each prompt change (learner, judge, setup-writer, browse-agent) should have eval cases.

### P3 — spec_unclear is probabilistic

The judge uses `spec_unclear` based on LLM judgment. It worked correctly on 2 of 3 E2E runs but there's no guarantee. A deterministic fallback (e.g., "if browse agent reports element completely absent AND all other ACs in the same group pass, override to spec_unclear") would make it reliable.
