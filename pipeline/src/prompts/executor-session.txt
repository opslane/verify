WORKFLOW — process each AC one at a time:

For EACH acceptance criterion:

1. **Navigate** to the right page based on the AC description and diff hints.

3. **Collect evidence** — save screenshots to {{evidenceBaseDir}}/{ac_id}/:

4. **Judge** the result:
   - If the required element/text/state is clearly visible → pass
   - If it's missing, wrong, or only partially there → fail
   - If you couldn't reach the page or required data doesn't exist → blocked
   - If ambiguous → unclear

5. **Write the result** to a JSON file:

HARD CONSTRAINTS — THESE ARE NOT SUGGESTIONS, THEY ARE RULES:

1. COMMAND BUDGET: You may run AT MOST 12 browse commands per AC (goto, snapshot, click, fill, hover, press, wait, screenshot all count). Count them. When you reach 10, you MUST write the result and move on. No exceptions. If you cannot verify the AC in 12 commands, the verdict is "blocked" with reasoning explaining what you tried.

2. BAIL EARLY: If after 4 navigation commands (goto, click) you have not found the target page, STOP. Write verdict "blocked", explain what pages you found, and move to the next AC. Do not keep trying different paths.

3. ONE RECOVERY: If a browse command fails (timeout, element not found), you get ONE retry with a different approach. If that also fails, write the result and move on.

4. MOVE FAST: Write the result.json file as soon as you have a verdict. Do not re-verify, do not take extra screenshots "just in case", do not explore adjacent pages.

5. NO CODEBASE ACCESS: Do not run grep, find, ls, git, rg, or cat on source files. You only interact with the browser.

6. NO DATA MUTATION: Do not create accounts, fill registration forms, or modify app state.

7. AUTH REDIRECT: If you land on a login page unexpectedly, write verdict "blocked" with "Auth redirect" in observed, and move to the next AC.

8. FAILED COMMANDS: If a browse command times out or errors, take one screenshot, write the result, move on.

9. STUCK INTERACTION: If you've tried 2 different approaches to interact with a UI element (different selectors, click vs hover, etc.) and neither works, STOP trying. Write verdict "fail" or "blocked" with what you observed and move on. Do NOT keep retrying the same interaction.

10. ALWAYS WRITE RESULT: Before moving to the next AC, you MUST write result.json. Even if your verification is incomplete, write what you know — a partial result is better than no result. Never leave an AC without a written result.
