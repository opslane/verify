WORKFLOW — process each AC one at a time:

For EACH acceptance criterion, follow this EXACT sequence:

2. **Navigate** to the right page based on the AC description and app routes.



HARD CONSTRAINTS — THESE ARE NOT SUGGESTIONS, THEY ARE RULES:

1. MARKERS ARE MANDATORY: Every AC MUST start with `echo "VERIFY_START|..."` and end with `echo "VERIFY_DONE|..."`. The supervisor uses these to track progress. If you skip markers, the supervisor cannot help you.

2. COMMAND BUDGET: Aim for AT MOST 12 browse commands per AC. When you reach 10, you MUST write the result and move on. The supervisor enforces a hard limit at 15 commands — if you exceed it, the supervisor writes a "blocked" verdict for you automatically.

3. PRECONDITION CHECK: After your first snapshot on the target page, if the required data is not visible, write "blocked" immediately and move on. Do NOT explore the entire app looking for data.

4. BAIL EARLY: If after 3 navigation commands (goto, click) you have not found the target page, write verdict "blocked" and move on.

5. ONE RECOVERY: If a browse command fails, you get ONE retry. Then write the result and move on.

6. NO CODEBASE ACCESS: Do not run grep, find, ls, git, rg, or cat on source files.

7. NO DATA MUTATION: Do not create accounts, fill forms that change state, or modify app data.

8. AUTH REDIRECT: If you land on a login page unexpectedly, write verdict "blocked" with "Auth redirect" in observed.

9. ALWAYS WRITE RESULT: Before moving to the next AC, you MUST write result.json. A partial result is better than no result.
