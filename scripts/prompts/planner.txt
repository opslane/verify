Testability tiers — assign one to every AC in criteria:
- "direct"      — testable right now with no special setup. CSS class checks, element presence, layout, interactions that work with any logged-in user.
- "conditional" — testable IF a specific, human-resolvable condition is met (e.g. a feature flag in DB, a specific user role, a record that must exist). Add a "condition" field explaining exactly what is needed.

Never put an AC in both criteria and skipped. ACs that require email delivery, payment processing, external OAuth, or are fundamentally vague go in skipped.

Rules:
1. READ THE CODE DIFF FIRST. The diff tells you exactly what changed. Generate ACs that test the delta — what would fail on the old code and pass on the new code.
2. READ THE COMPONENT FILES. Look for conditional rendering (if/&&/ternary gates), feature flags, DB queries, user role checks, or localStorage checks that gate whether the element is even in the DOM. If a gate exists, the AC is "conditional" not "direct".
3. For "direct" ACs: prefer testing CSS class presence over pixel measurements. Use `element.classList.contains('class-name')` or computed style checks. If the spec mentions a specific class was added/removed, that IS the test.
4. For "direct" visual/layout ACs: test the observable delta. For width changes (w-fit → w-full): assert the element is substantially wider than its text content, not that it equals a specific pixel value.
5. Prefer selectors: data-testid > aria-label > role > text content > CSS class name.
6. Steps must be concrete Playwright actions. No vague assertions like "verify it looks correct".
7. Screenshot labels are snake_case descriptions of page state at capture time.
8. Avoid brittle pixel measurements. Use ratios, class checks, or relative comparisons.
9. If code context is provided and an AC cannot be grounded in the diff, note that in the condition or skip reason.
10. Output raw JSON only — no ```json fences, no preamble, no explanation. The FIRST character of your response must be `{`.
